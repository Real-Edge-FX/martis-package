# Internationalisation

> Adding, overriding and extending locales in Martis.

Martis ships with three bundled locales (`en`, `pt_PT`, `pt_BR`) and exposes them in the user preferences panel. The system has three layers — **PHP lang files** (server strings), **i18next resources** (client strings), and the **user preferences** (per-user locale persistence).

---

## How a locale reaches the UI

```
┌─────────────────┐
│  PHP __() keys  │  ← resources/lang/{locale}/*.php
└────────┬────────┘
         │ (TranslationsController /api/translations/{locale})
         ▼
┌─────────────────┐
│  i18next client │  ← loaded at boot, mutated by PreferencesMenu
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  React t() keys │
└─────────────────┘

┌─────────────────┐
│ User preference │  ← PUT /api/preferences { locale }
└────────┬────────┘
         │
         ▼
┌──────────────────────────────┐
│  martis.locale middleware    │
│  → app()->setLocale()         │
│  → __() returns right string  │
└──────────────────────────────┘
```

Two fetches happen when the user changes language in the preferences panel:

1. `PUT /api/preferences` persists the new locale.
2. `loadLocale()` re-fetches `/api/translations/{locale}`, merges the resources into i18next, then calls `queryClient.invalidateQueries()` so every server-side label (navigation, lenses, resource titles) re-fetches under the new locale.

**Order matters** — the PUT is awaited first so React Query invalidation reads the fresh preference from the DB.

### Server-side cache keys include the locale

Lens responses (`LensController`) and metric results (`Metric::resolve()`) are cached server-side for up to `cacheTtl()` seconds. Both cache keys now include `app()->getLocale()` so column headers, summary labels, and metric bucket names pick up the new locale immediately after a language switch — without waiting for the TTL to expire. Consumer-built controllers that wrap responses in `Cache::remember()` should follow the same pattern when the cached payload contains `__()`-translated text.

## Bundled locales

| Code | Label in UI | Ships with | Fallback |
|------|-------------|------------|----------|
| `en` | English | Package defaults | — |
| `pt_PT` | Português (PT) | Package defaults | `en` |
| `pt_BR` | Português (BR) | Package defaults | `en` |

Labels are read from `config('martis.preferences.locale_labels')`. The code itself is always what's persisted and sent to the API.

## Adding a new locale (e.g. Spanish)

### 1. Ship translation files

Create `resources/lang/es/*.php` in your app mirroring the package structure:

```
resources/lang/es/
├── actions.php
├── auth.php
├── martis.php
├── messages.php
├── metrics.php
├── navigation.php
├── profile.php
└── resources.php
```

The fastest path is copying from `lang/vendor/martis/en/` (after `vendor:publish --tag=martis-lang`) and translating values.

If you want to **override just a subset** of strings without shipping full files, publish the vendor lang and edit the files:

```bash
php artisan vendor:publish --tag=martis-lang
# edit resources/lang/vendor/martis/es/*.php
```

Resolution order in `TranslationsController` (each layer applied with `array_replace_recursive`, so later writes win per key):

1. **Fallback chain** — every locale in `config('martis.locales.fallback_chain')` (default `['en']`) seeds the payload first.
2. **Package defaults** — `resources/lang/{locale}` shipped by Martis.
3. **App namespaces** — host-app files in `lang/<locale>/<ns>.php` for every namespace listed in `config('martis.locales.app_namespaces')`.
4. **Consumer publishes** — `resources/lang/vendor/martis/{locale}` (your app-level overrides). Wins over everything else.

`array_replace_recursive` merges per-key, so dropping a single-key file in `lang/vendor/martis/<locale>/actions.php` overrides only that key. The other keys keep flowing from the package defaults — no need to copy the whole namespace.

### 2. Register the locale

```php
// config/martis.php

'preferences' => [
    'locales' => ['en', 'pt_PT', 'pt_BR', 'es'],
    'locale_labels' => [
        'en' => 'English',
        'pt_PT' => 'Português (PT)',
        'pt_BR' => 'Português (BR)',
        'es' => 'Español',
    ],
],
```

### 3. Translate your own resources

The package only handles its own strings. For app resources (like a `ClientResource`), use `__('app-keys.clients')` in `label()`/`group()`/field labels and add the keys to your app's own lang files.

```php
// app/Martis/Resources/ProductResource.php

public static function label(): string
{
    return __('my-app.products');  // your own lang namespace
}
```

### 4. Clear caches after adding

```bash
php artisan config:clear
php artisan cache:clear
```

The language dropdown picks up new entries on the next hard reload.

## ⭐ Extensibility differentials

Three knobs in `config/martis.php`'s `'locales'` section let consumer apps extend Martis's translation pipeline without forking the package. Defaults preserve the historical behaviour exactly — these are purely opt-in.

### 1. Per-key deep merge (always-on)

Consumer overrides published to `lang/vendor/martis/<locale>/` are merged into the package defaults at the **key level**, not the file level. You only ship the keys you want to change.

```php
// resources/lang/vendor/martis/en/actions.php
return [
    'create' => 'Add new',  // overrides Martis's 'Create'
];
```

The rest of `actions.*` keeps flowing from the package's own `actions.php`. No need to copy the whole file, no risk of falling behind when the package adds new keys in a future release.

### 2. App namespaces auto-merged

By default, `/api/translations/<locale>` returns the package's own namespaces (`actions`, `auth`, `martis`, `messages`, `metrics`, `navigation`, `profile`, `resources`). To surface your own translation files in the same payload — without writing a parallel loader on the frontend — list them:

```php
// config/martis.php
'locales' => [
    'app_namespaces' => ['app', 'billing'],
    // ...
],
```

```php
// resources/lang/en/app.php
return [
    'tagline' => 'Run the business',
    'support_email' => 'support@example.com',
];
```

Now your React code can call `t('app:tagline')` (or `useTranslation('app')`) and get the value, with the same publish + deep-merge story as Martis's own namespaces. Names are validated against `^[a-z][a-z0-9_]*$` so a malformed entry can't escape the locale folder.

The `MARTIS_APP_LOCALE_NAMESPACES` env var accepts a comma-separated list as a shortcut: `MARTIS_APP_LOCALE_NAMESPACES=app,billing`.

### 3. Configurable fallback chain

The default chain is `['en']`, matching the historical "fall back to English when missing" behaviour. Multi-step chains are useful for related locales:

```php
// config/martis.php
'locales' => [
    'fallback_chain' => ['pt_BR', 'en'],
],
```

With this config, a `pt_PT` request seeds missing keys from `pt_BR` first, then falls back to `en` if `pt_BR` is also missing the key. The requested locale is automatically filtered out of the chain so it never seeds itself.

Equivalent env: `MARTIS_LOCALE_FALLBACK_CHAIN=pt_BR,en`.

## Locale formats

Laravel supports both flat (`es`) and underscore-variant (`pt_BR`, `es_MX`) codes. Martis preserves whatever code you put in config — no automatic case-folding.

The `TranslationsController` normalises inbound locale slugs: `pt_br` → `pt_BR`, `EN_US` → `en_US`. Unknown codes fall back through `martis.locales.fallback_chain` (default `['en']`).

## Changing locale at runtime

**From the user menu**: Preferences panel → Language dropdown. Persists to DB, refetches resources, invalidates React Query cache.

**From code**:

```ts
import { loadLocale } from '@/lib/i18n'
import { usePreferences } from '@/contexts/PreferencesContext'

const { update } = usePreferences()
await update({ locale: 'es' })
await loadLocale('es')
```

---

## Backend locale (server-rendered strings)

The `martis.locale` middleware runs inside the authenticated group and calls `app()->setLocale($userLocale)` before controllers run. Every `__()` call inside controllers (navigation labels, validation messages, resource titles) therefore returns the user's chosen locale.

The middleware is tolerant — missing table, missing user, or invalid code all fall through without errors. The app keeps `config('app.locale')` as the ultimate default.
